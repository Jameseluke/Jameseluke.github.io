Risk Assessment

James Luke
2017-05-04

In performing a risk assessment of our website the main thing that we as a group had to consider were the security risks and vulnerabilities that we had yet to take into consideration. Looking through a majority of our back-end code we realised however that we had consciously thought about these problems throughout the creation of our website and so were relatively well protected from most attacks.

In login and registration requests, validation is occurring on both the front and back ends to ensure that for a majority of requests, correct details will be provided – but on the off chance that users may attempt to circumnavigate our system, backend validation will prevent any nefarious attacks.

Similarly, sanitising input forms across all pages to ensure that they cannot be manipulated to perform malicious actions against our forms or databases has also been put into place reducing the possibility of any kind of fake form attack on our website.

A majority of work is still being completed to prevent staple attacks such as cookie theft, cross-site scripting and cross-site request forgery.

Risk Trigger Likelihood Impact Contingency plan
Risk, Trigger Likelihood Impact Contingency Plan
Group members not completing set work Falling behind in timeline plan Moderate High Reduce scope of project to core components
NoSQL injection of user database Leaked database found online Low High Warn all users of data breach, increase security measures
Database loss Missing database Low High Revert to last database backup
DDOS Server crashes or site is really slow Medium Medium DDOS prevention
URL Manipulation Can still access privileged pages without sign-in Moderate Low Ensure certificate so only logged in users can access these pages
Cross-site scripting Certain inputs still able to be modified Moderate High Sanitise all input forms

In further increasing the security of our website though, these risks have to be mitigated. Some of these risks are simply unavoidable such as database loss – if it happens we simply have to deal with it and revert to a previous backup to continue with a functioning website.
Others are preventable and are being accounted for as development on the website continues.

NoSQL injections are preventable by sanitising user input to prevent any arbitrary javascript being run on the Mongo database.
Likewise, URL manipulation can also be prevented by ensuring that privileged pages requiring sign-in details can only be viewed by those that have actually signed in. Keeping track of logged in users and noting if they have received a legitimate authentication through our server (likely via passport) removes this security issue entirely.
